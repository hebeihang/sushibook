# SushiBook (SushiML) 语法评审

> 评审依据：**实现源码**（`src/sushiml/parser.ts`、`types.ts`、`bridge.ts`），非 README。
> 对照对象：**Kiny DSL 规范 v0.1.0 草案**（Ink 的精简版互动叙事 DSL）。
> 评审日期：2026-07-15

---

## 0. 结论先行

SushiBook 语法整体**可用**，且有 Kiny 没有的真亮点（`[[标记词]]` 悬停注解 + 词语级动效、场景 `---` frontmatter 表现元数据、更细的静态检查）。但确实有几个实打实的问题，核心症结是：**把"叙事内容 + 渲染意图 + 运行时逻辑"全塞进了一个 `{ }`。** Kiny 在语法纯净度上明显更干净，代价是放弃了 SushiBook 那些表现力特性。

本文分四部分：① 已证实的问题（带行号）；② 与 Kiny 逐项对比；③ 修改建议（按性价比）；④ 前瞻思考——**内容层 / 渲染层分离**。

---

## 一、已证实的问题

### 1.1 `{ }` 一身兼四职，消歧靠白名单且很脆

`{ }` 在 SushiML 里有四种含义：

| 含义 | 触发条件 | 示例 |
|---|---|---|
| 句尾效果指令 | 行尾且 key 全命中白名单 | `{typewriter: 60ms}` |
| 词语指令 | 紧跟 `]]` | `[[船]{enter/color}` |
| 行内 JS 插值 | 任意位置 | `{gold}` |
| 变体函数 | `seq:/cycle:/once:/shuffle:` | `{seq:a\|b\|c}` |

消歧规则在 `parser.ts:507-512`：

```ts
const directives = parseDirectiveString(match[1]);
const keys = Object.keys(directives);
const allKnown = keys.length > 0 && keys.every((k) => SENTENCE_DIRECTIVE_KEYS.has(k));
if (!allKnown) {
  return { text: line, directives: {} };   // 没认出 → 整段当普通文本保留
}
```

即：只有花括号内**所有 key 都命中白名单**（`typewriter/pause/pause-before/pause-after/flash`，见 `types.ts:45-49`）才算效果指令，否则整段 `{...}` 当作字面文本留在输出里。

**踩坑示例：**
- 拼错 `{typewritter: 60ms}`（多一个 t）→ 不在白名单 → 被当 JS 表达式/字面文本 → 输出异常且不报错。
- 把效果指令写在句中（非句尾）→ 同样走不到白名单分支 → 变垃圾文本。

> 这是与 Kiny「`{ }` 只有一种含义（JS 表达式求值）」原则的硬冲突点。

### 1.2 变体"锁步"推进（最隐蔽，建议必改）

`bridge.ts:633-649` 的 `resolveToken` 用 `this.visits.get(sceneId)`（场景访问次数）驱动 `seq/cycle/once`：

```ts
const visit = Math.max(this.visits.get(sceneId) ?? 1, 1);   // bridge.ts:637
// ...
case 'seq':
  return token.items[Math.min(visit - 1, n - 1)];            // 靠 visit
case 'cycle':
  return token.items[(visit - 1) % n];
case 'once':
  return visit - 1 < n ? token.items[visit - 1] : '';
```

而 `variantIndex`（每个调用点的独立序号，已在 `parser.ts:594` 算好）**只用于 `shuffle` 的随机种子**，不参与 `seq/cycle/once` 的推进。

**后果**：同一场景内所有 `seq/cycle/once` 共享一个"场景访问次数"计数器、齐步前进。例如场景 `雾港` 访问 3 次，里面两个 `{seq:甲|乙|丙}` 和 `{seq:一|二|三}` 会**同步**从第 1 项走到第 3 项，彼此无法独立推进。

**Kiny 对照**：Kiny 每个 `seq()` / `cycle()` / `once()` 调用点各自按源码位置维护独立计数，互不干扰。

**修复很小**：把变体状态按 `variantIndex`（每调用点）键控，而非依赖共享的 `visit`；同时保留"场景重访才推进"的语义（`visit` 仍作为推进的触发条件，但每个调用点记录自己的进度）。约一行级改动即可追平 Kiny。

### 1.3 没有多行 JS、没有自定义函数

只有单行 `~ 语句`（`~ let gold = 10`）。复杂逻辑（背包、战斗数值、循环）只能一行一行堆。Kiny 有 `~~~ ... ~~~` 块（`§7.4`）+ JS 函数声明（`§12`），这块 SushiBook 是空白。

### 1.4 `>` 家族视觉冲突

- `>` = 分支体层级
- `>>` = 粘性选项
- `->` = 跳转

三种 `>` 系符号并存。`parser.ts:243` 注释明示：**嵌套选项必须写成 `> >> 选项`**。扫读时 `>>` 看着像"双跳转"，容易和 `->` 混淆。Kiny 用 `*`/`+` 表示选项、把 `>` 专留给分支体，一眼能分。

**建议**：SushiBook 把 `>>`（粘性）改成 `+`，与 Kiny 对齐，`>` 只做分支体。

### 1.5 变体项里不能含 `|`

`parser.ts:593` 直接 `variant[2].split('|')`，变体文本里出现 `|` 会被拆成两项，且无转义/引号机制。Kiny 用 `seq("a|b")` 引号包裹，天然安全。

### 1.6 小坑

- **frontmatter 是玩具级 YAML 解析**：`parseYamlLike` 只按第一个 `:` 切，全当字符串，无类型、无引号；`---` 格式稍有偏差就静默返回空。
- **词语指令漏写 `]]`**：若 `[[词` 直接写成句尾 `{color:…}` → 不在句尾白名单 → 整段当字面文本，不报错。

## 二、与 Kiny 逐项对比

| 维度 | SushiBook (SushiML) | Kiny | 谁更优 |
|---|---|---|---|
| `{ }` 含义 | 效果/词语指令/插值/变体 四合一 | 仅 JS 表达式求值 | **Kiny** |
| 变体推进 | 靠场景访问数、同场景锁步 | 每调用点独立计数 | **Kiny**（SushiBook 改一行可追平）|
| 多行逻辑/函数 | 无 | `~~~` 块 + JS 函数 | **Kiny** |
| 选项符号 | `*` 一次性 / `>>` 粘性 | `*` 一次性 / `+` 粘性 | 平（SushiBook 符号更乱）|
| 分支体 | `>` 层级（与 `>>` 冲突）| `>` 层级（专属）| **Kiny** |
| 条件链 | `@if/@elif/@else` | 同 | 平 |
| 跳转 / END | `-> 目标` / `-> END` | `-> 目标` / `-> END` / `-> DONE` | 平 |
| 选项标签 | `(label)` 自动计数 | 同（`§5.5`）| 平 |
| 粘连 | `<>` 句尾 | `<>` 句尾（明确跨跳转）| 平 |
| 中文 ID | 允许（`\S+`）| 允许 | 平 |
| 变体文本含 `\|` | 会拆坏 | 引号安全 | **Kiny** |
| **标记词 + 注解 + 词语动效** | `[[词]]` + `\|` 注解 + `{enter/color}` | **无** | **SushiBook** |
| **场景表现元数据** | `---` frontmatter (mood/enter/speed) | 无（靠 `@` 命令/渲染器）| **SushiBook** |
| 静态检查 | 悬空链接 + 三类死胡同 + 入链统计 | 重名 + 死胡同警告 | **SushiBook** 更细 |
| `@bg_show` 入参 | 接受裸 CSS 渐变（容错好）| 仅资源路径 | **SushiBook** 更灵活 |

**小结**：SushiBook 赢在"表现力与可检查性"，Kiny 赢在"语法纯净度与逻辑表达力"。两者在结构/流程层高度同构（节点、选项、分支体、跳转、条件、粘连、标签几乎一一对应）。

## 三、修改建议（按性价比排序）

1. **修变体锁步**（必做）：`bridge.ts:637` 把共享 `visit` 换成按 `variantIndex` 键控的每调用点进度表。一行级改动，直接追平 Kiny 的"每调用点独立"。
2. **`>>` → `+`**：粘性选项改 `+`，`>` 专用于分支体，消除视觉冲突、与 Kiny 同构。
3. **变体加引号/转义**：`{seq:"a|b"}` 或至少支持 `\|` 转义，避免 `|` 拆项。
4. **`{ }` 是否收口**（取舍点）：它服务于 `[[标记词]]` 这个 Kiny 没有的杀手特性，所以"词语指令走 `{ }`"有正当理由；真正该收的是**句尾效果指令**——可考虑换独立符号（如 `{@ typewriter: 60ms}`），让行内 `{ }` 回归纯 JS，消歧更稳。详见第四节的分层方案。

## 四、前瞻思考：内容层 / 渲染层分离

### 4.1 问题的根源不在"语法脏"，而在"语义未分层"

回看 §1.1，所有坑都来自同一个事实：**SushiML 用同一套 `{ }` 同时承载了三种不同性质的语义——**

| 性质 | 例子 | 应该由谁消费 |
|---|---|---|
| **叙事内容** | 文本、`[[标记词]]` 的标注 | 故事模型、校验器、TTS/终端渲染器 |
| **渲染意图** | `{typewriter: 60ms}`、`{flash}`、`{color}` | 网页粒子渲染器 |
| **运行时逻辑** | `{gold}`、`{seq:a\|b}`、条件 `{...}` | 解释器/虚拟机 |

它们被压进同一个花括号，解析器只能用"白名单猜"来消歧（`parser.ts:509`），猜错就静默退化成字面文本。这不是加几个特例能根治的——**根因是内容、表现、逻辑三种关注点没有在语法/IR 层面分开。**

### 4.2 三层模型

建议把 SushiBook 的运行时拆成三层：

```
┌─────────────────────────────────────────────┐
│  Layer 3  渲染器 (Renderer)                    │
│  Web 粒子 / 终端 / TTS / 音频 —— 可插拔、可换肤 │
├─────────────────────────────────────────────┤
│  Layer 2  表现层 (Presentation / Style)        │
│  场景级: mood/enter/background/bgm (frontmatter)│
│  词级:   typewriter/pause/flash/color/size     │
│  —— 只描述"怎么呈现"，不碰"说什么"            │
├─────────────────────────────────────────────┤
│  Layer 1  内容层 (Content / Story AST)         │
│  文本 / 节点 / 选项 / 跳转 / 条件 / JS逻辑 /     │
│  变体逻辑 / [[标记词]]标注 —— 纯数据、可移植     │
└─────────────────────────────────────────────┘
```

- **Layer 1 内容层**：纯故事数据（AST）。可被校验器静态检查、可被任意渲染器消费、可零渲染导出（给终端/TTS/语音版）。这一层的 `{ }` 应**只有一种含义 = JS 表达式求值**（与 Kiny 对齐）。
- **Layer 2 表现层**：只描述"怎么呈现"——场景级已通过 `---` frontmatter 落到了这层（很好）；词级效果指令（`typewriter/flash/color`）应明确归属这层，而非混进内容。
- **Layer 3 渲染器**：消费 L1+L2，产生实际输出。换渲染器 = 换皮肤，故事逻辑不动。

### 4.3 关键洞察：分离 ≠ 两个文件

作者喜欢把 `[[船]{flash}` 直接写在词旁边——这种内联人体工学好，不该丢掉。**分离可以在 IR（中间表示）层面做，而不必在源文件层面做。**

即：源文件仍是作者友好的单文件，但 **parser 的输出应是一个判别联合（discriminated union）**，而非靠运行时白名单猜：

```ts
type SushiToken =
  | { kind: 'text';     value: string }
  | { kind: 'mark';     word: string; note?: string; /* 内容层数据 */ }
  | { kind: 'expr';     code: string }                 // 内容/逻辑层：纯 JS
  | { kind: 'variant';  fn: 'seq'|'cycle'|'once'|'shuffle'; items: string[]; id: number }
  | { kind: 'fx';       directive: SentenceDirective } // 表现层：渲染意图
```

解析阶段就**明确打标**：每个 `{ }` 是 `expr` 还是 `fx` 还是 `variant`，不再靠 `SENTENCE_DIRECTIVE_KEYS` 事后猜。渲染器只读 `fx`，内容层只产 `text/mark/expr/variant`。这样：
- §1.1 的静默退化消失（猜错 → 编译期报错，而非输出垃圾文本）；
- 内容层可整体导出给非粒子渲染器；
- 表现层可独立成"样式表"或保持内联，二者不冲突。

### 4.4 与 Kiny 的对照

Kiny 其实已经做了**部分**分离：
- 内容层 = `.kin` 语法（叙事 + JS 逻辑）；
- 渲染层 = `@bg_show()` 等命令 + 宿主阅读器的样式。

但 Kiny 的 `@` 命令仍是**内联渲染指令**，没有独立的 IR 分层；它只是把渲染推出了 `{ }`，而非把内容/表现彻底解耦。理想态是：**同一份内容 AST 可以零渲染导出**（给终端/TTS），渲染指令是可选的叠加层——这正是 4.2/4.3 想补上的那一步。

### 4.5 落地路径（低侵入）

1. **短期（改 parser 输出）**：把 §1.1 的白名单消歧前移为"解析期打标"，产出上面那个判别联合。风险低、收益高。
2. **中期（修变体 + 选项符号）**：落实 §三.1、§三.2，顺便让变体状态按 `variantIndex` 键控。
3. **长期（表现层外置可选）**：frontmatter 已就位；词级 `fx` 指令可保留内联（人体工学），但允许一份外部 `*.sushi-style` 覆盖/补充，实现"换肤不碰故事"。

> 一句话：**SushiBook 不必在"表现力"和"纯净度"之间二选一——把关注点在 IR 层面分层，就能两者兼得。**
